# InternPilot 二次开发项目计划文档

> 项目定位：在现有 InternPilot 求职助手基础上，扩展为一个“可采集、可管理、可分析、可推荐、可生成材料”的 AI 求职投递信息管理与岗位智能分析系统。

---

## 1. 项目背景与现有基础

### 1.1 现有项目基础

本项目拟基于开源项目 InternPilot 进行二次开发。InternPilot 当前已经具备较完整的个人求职助手雏形，主要功能包括：

- 求职控制台：展示岗位总览、高匹配岗位、投递进度统计；
- 岗位管理：支持粘贴 JD 自动识别、手动导入、岗位编辑、看板式投递进度管理；
- 智能匹配：基于个人经历与岗位信息进行双向关键词匹配，并调用 AI 生成推荐理由；
- 个人背景管理：支持教育背景、项目经历、实习经历等信息录入，并可从简历中提取关键词；
- 简历生成：根据岗位和个人经历生成针对性背景描述与 bullet points；
- 面试准备：根据岗位生成面试题、知识点，并提供岗位隔离的 AI 问答助手；
- 浏览器插件：在招聘页面一键剪藏岗位信息，并保存到本地系统。

现有技术栈主要包括：

- 后端：Python + FastAPI；
- 前端：React；
- 数据库：SQLite；
- AI 调用：DeepSeek API，兼容 OpenAI 调用格式；
- 浏览器插件：Chrome Extension Manifest V3。

### 1.2 当前项目存在的不足

虽然 InternPilot 已经覆盖了求职管理的基本闭环，但如果作为课程项目或数据挖掘/AI 应用项目，目前还存在几个明显不足：

1. **数据来源偏被动**  
   现有系统主要依赖用户手动粘贴 JD 或使用浏览器插件剪藏岗位，缺少主动采集主流招聘网站岗位信息的能力。

2. **AI 功能偏辅助，智能程度不足**  
   当前 AI 主要用于 JD 解析、推荐理由、简历生成和面试准备，更多是文本生成工具，尚未形成更完整的智能分析系统。

3. **缺少明确的数据挖掘/模型训练环节**  
   如果课程要求体现数据挖掘或机器学习，单纯调用大模型 API 不够充分，需要加入岗位分类、匹配预测、薪资/要求分析、岗位趋势挖掘等可训练模型。

4. **RAG 使用场景不够明确**  
   现有 AI 问答虽然会结合岗位 JD 和个人经历上下文，但还没有形成结构化的知识库检索增强生成机制。

5. **岗位管理与求职决策之间的连接还可以加强**  
   系统不仅应记录岗位，还应帮助用户判断：哪些岗位值得投、为什么值得投、如何修改简历、如何准备面试、后续投递策略如何调整。

## 2. 项目目标

### 2.1 总体目标

在 InternPilot 现有功能基础上，构建一个面向个人求职者的 AI 求职投递信息管理与岗位智能分析系统。系统能够自动或半自动采集岗位信息，构建个人求职知识库，利用数据挖掘和机器学习模型分析岗位与用户经历的匹配程度，并通过 RAG 与大模型生成个性化求职建议、简历优化内容和面试准备材料。

### 2.2 核心目标

1. **降低岗位信息录入成本**  
   支持从主流招聘网站自动采集岗位信息，同时保留链接粘贴、文本上传、浏览器插件剪藏等方式。

2. **增强岗位分析能力**  
   对岗位进行自动分类、技能标签抽取、岗位要求结构化、匹配度计算和投递优先级排序。

3. **体现数据挖掘和模型训练**  
   在传统关键词匹配之外，引入可训练的岗位分类模型、匹配度预测模型或岗位聚类分析模型。

4. **构建 RAG 求职知识库**  
   将用户简历、项目经历、过往投递记录、岗位 JD、面试笔记等内容向量化存储，使 AI 回答能够基于用户真实资料和岗位上下文生成。

5. **强化 AI 的求职决策角色**  
   让 AI 不只是生成文本，而是参与岗位筛选、岗位解释、简历改写、面试准备、投递策略建议和求职复盘。

## 4. 用户需求分析

### 4.1 目标用户

主要面向正在找实习、校招、社招或转行机会的学生与年轻求职者，尤其适合同时投递多个岗位、需要管理大量 JD、需要针对不同岗位修改简历和准备面试的人群。

### 4.2 用户痛点

1. 招聘网站分散，岗位信息需要反复搜索和复制；
2. 投递岗位越来越多后，容易忘记岗位来源、投递阶段和跟进时间；
3. 每个岗位 JD 很长，用户很难快速判断是否值得投；
4. 简历需要针对不同岗位微调，但人工修改成本高；
5. 面试准备缺少针对性，往往不知道该重点准备哪些经历和知识点；
6. 用户无法从历史投递数据中总结自己适合什么类型岗位、哪些岗位转化率更高。

### 4.3 需求转化

| 用户痛点 | 系统功能 |
|---|---|
| 岗位信息分散 | 招聘网站采集、链接解析、浏览器插件剪藏 |
| 投递信息混乱 | 岗位看板、投递阶段、提醒与备注 |
| JD 难以快速理解 | JD 结构化解析、技能标签抽取、岗位摘要 |
| 不知道岗位是否值得投 | 匹配度计算、投递优先级、推荐理由 |
| 简历修改成本高 | 针对岗位的简历 bullet 生成与优化 |
| 面试准备不够针对 | 岗位面试题、知识点、追问模拟 |
| 缺少求职复盘 | 投递数据统计、岗位类型聚类、转化率分析 |

## 5. 系统总体设计

### 5.1 总体架构

系统整体可以分为六层：

1. **数据采集层**  
   负责从招聘网站、岗位链接、用户上传文件、浏览器插件中获取岗位数据。

2. **数据处理层**  
   对岗位 JD 进行清洗、去重、字段抽取、技能标签识别和结构化存储。

3. **数据挖掘与模型层**  
   使用机器学习模型进行岗位分类、岗位聚类、匹配度预测和技能需求分析。

4. **RAG 知识库层**  
   将岗位 JD、用户简历、个人经历、投递记录、面试笔记等内容切片、向量化并建立检索系统。

5. **AI 应用层**  
   基于大模型实现岗位解读、推荐理由、简历优化、面试准备、投递策略建议等功能。

6. **前端交互层**  
   提供岗位看板、岗位详情、推荐列表、AI 聊天、简历生成、数据分析仪表盘等界面。

### 5.2 技术路线

| 模块 | 技术方案 |
|---|---|
| 后端服务 | FastAPI |
| 前端页面 | React |
| 本地数据库 | SQLite，后续可扩展 PostgreSQL |
| 招聘信息采集 | requests / Playwright / 浏览器插件 / 用户粘贴链接 |
| 文本解析 | LLM + 规则抽取 + 正则清洗 |
| 向量数据库 | FAISS 或 Chroma |
| Embedding | bge-small-zh / text-embedding-3-small / 其他中文向量模型 |
| 机器学习 | scikit-learn / lightgbm / sentence-transformers |
| 大模型调用 | DeepSeek API 或 OpenAI 兼容接口 |
| 可视化 | ECharts / Recharts |

## 6. 功能模块规划

## 6.1 模块一：岗位信息采集模块

### 6.1.1 功能目标

在原有“粘贴 JD”和“浏览器插件剪藏”的基础上，增加主动采集和链接解析能力，降低用户录入岗位信息的成本。

### 6.1.2 功能设计

1. **主流招聘网站岗位采集**
   - 支持用户输入关键词、城市、岗位类型、经验要求等筛选条件；
   - 从指定招聘网站采集岗位标题、公司名称、地点、薪资、岗位描述、岗位要求、链接等字段；
   - 支持定时采集或手动刷新；
   - 对采集结果进行去重和更新。

2. **岗位链接解析**
   - 用户粘贴岗位详情页链接；
   - 系统自动抓取页面文本；
   - 调用 JD 解析模块抽取结构化字段；
   - 用户确认后保存到岗位库。

3. **文本/文件上传导入**
   - 支持上传 txt、docx、pdf 或粘贴大段岗位信息；
   - 系统自动识别其中包含的一个或多个岗位；
   - 拆分后写入岗位库。

4. **浏览器插件增强**
   - 保留原有剪藏能力；
   - 增加“采集后自动匹配”“保存到指定投递阶段”“标记来源平台”等功能。

### 6.1.3 注意事项

由于招聘网站存在反爬限制，项目实现时可以采用“课程项目可演示”的渐进方案：

- 第一阶段：支持用户粘贴链接和插件剪藏；
- 第二阶段：对公开可访问页面进行解析；
- 第三阶段：对固定样例页面或模拟招聘网站进行批量采集；
- 第四阶段：视情况支持真实网站的低频采集。

## 6.2 模块二：岗位结构化解析模块

### 6.2.1 功能目标

将非结构化 JD 文本转化为可计算、可检索、可分析的结构化数据。

### 6.2.2 抽取字段

| 字段 | 示例 |
|---|---|
| 岗位名称 | 产品经理实习生 |
| 公司名称 | 某科技公司 |
| 工作地点 | 北京 / 上海 / 远程 |
| 岗位类型 | 产品 / 运营 / 算法 / 后端 / 销售 |
| 学历要求 | 本科及以上 / 硕士优先 |
| 经验要求 | 不限 / 1-3 年 / 有实习经历优先 |
| 技能要求 | SQL、Python、数据分析、Axure |
| 软性要求 | 沟通能力、逻辑能力、学习能力 |
| 工作内容 | 用户调研、需求分析、数据看板 |
| 加分项 | AI 产品经验、ToB 项目经验 |
| 薪资范围 | 150-200/天 或 15k-25k |
| 来源平台 | Boss / 实习僧 / 拉勾 / 手动导入 |
| 原始链接 | 岗位详情页链接 |

### 6.2.3 技术实现

- 使用规则抽取处理明显字段，如城市、薪资、学历；
- 使用 LLM 抽取复杂语义字段，如职责、要求、加分项；
- 使用技能词典和实体识别抽取技能标签；
- 将抽取结果保存到数据库，同时保留原始 JD 文本，便于溯源。

## 6.3 模块三：数据挖掘与模型训练模块

该模块是项目区别于普通 AI 工具的关键，也是体现课程中“数据挖掘”要求的核心部分。

### 6.3.1 子任务一：岗位类型分类模型

#### 任务目标

根据岗位标题和 JD 内容，自动判断岗位类别，例如：

- 产品经理；
- 数据分析；
- 算法工程；
- 后端开发；
- 前端开发；
- ToB 销售；
- 市场运营；
- 咨询/项目管理。

#### 数据来源

- 系统采集的岗位 JD；
- 用户手动导入的岗位；
- 人工标注的岗位类别。

#### 建模方案

基础方案：

- 文本特征：TF-IDF；
- 模型：Logistic Regression / SVM / Random Forest；
- 指标：Accuracy、Macro-F1、Confusion Matrix。

增强方案：

- 文本向量：Sentence-BERT / bge 中文向量；
- 模型：LightGBM / MLP / 微调小模型；
- 指标：Macro-F1、Weighted-F1。

#### 产出

- 一个可训练、可评估、可保存的岗位分类模型；
- 前端岗位详情页展示“模型预测岗位类别”；
- 用户可手动修正类别，修正数据反哺模型。

---

### 6.3.2 子任务二：岗位匹配度预测模型

#### 任务目标

根据用户经历与岗位要求，预测该岗位对用户的匹配程度，并输出投递优先级。

#### 特征设计

| 特征类型 | 示例 |
|---|---|
| 技能重合度 | JD 技能标签与用户经历技能标签的 Jaccard 相似度 |
| 语义相似度 | 岗位描述向量与用户经历向量的余弦相似度 |
| 学历/地点匹配 | 是否满足学历、城市、实习时长要求 |
| 经验相关度 | 用户项目/实习经历与岗位职责的相似度 |
| 偏好匹配 | 是否符合用户目标岗位、行业、城市、薪资偏好 |
| 历史反馈 | 用户是否收藏、投递、拒绝、面试通过 |

#### 标签来源

初期可以使用弱监督标签：

- 用户收藏/标记“想投”的岗位 = 高匹配；
- 用户标记“不适合”的岗位 = 低匹配；
- 用户实际投递并进入面试 = 正样本；
- 长期未处理或删除 = 低优先级样本。

#### 建模方案

- 初版：规则加权打分；
- 中期：Logistic Regression / XGBoost / LightGBM；
- 后期：结合语义向量的 Learning to Rank 模型。

#### 产出

- 岗位匹配分数；
- 投递优先级：高 / 中 / 低；
- 影响因素解释：技能匹配、经历匹配、偏好匹配、短板项。

---

### 6.3.3 子任务三：岗位聚类与市场趋势分析

#### 任务目标

对采集到的岗位进行无监督聚类，帮助用户理解当前岗位市场中的主要需求类型。

#### 方法设计

- 使用岗位 JD 的文本向量；
- 使用 K-Means / HDBSCAN 进行聚类；
- 对每个簇提取高频关键词和代表岗位；
- 生成岗位市场分析报告。

#### 分析内容

- 当前岗位主要分为哪些类型；
- 不同岗位类型最常见技能要求；
- 哪些岗位与用户经历最接近；
- 哪些技能是用户需要补足的高频短板；
- 不同岗位类型的薪资、城市、学历要求差异。

#### 产出

- 岗位聚类可视化；
- 技能词云或技能频次图；
- 岗位类型趋势分析；
- 用户个人求职方向建议。

## 6.4 模块四：RAG 求职知识库模块

### 6.4.1 为什么需要 RAG

本项目中 RAG 的价值不在于回答通用百科问题，而在于让 AI 能够基于用户自己的求职资料和岗位库进行回答。这样可以避免大模型凭空生成不符合用户经历的内容，也能让简历优化和面试准备更有针对性。

### 6.4.2 知识库内容

| 知识类型 | 内容示例 |
|---|---|
| 用户简历 | 教育背景、项目经历、实习经历、技能 |
| 岗位 JD | 职责、要求、加分项、公司信息 |
| 投递记录 | 投递阶段、时间、反馈、备注 |
| 面试资料 | 面试题、回答草稿、复盘笔记 |
| 生成历史 | 简历版本、AI 推荐理由、面试准备内容 |

### 6.4.3 RAG 流程

1. 文档切片：将简历、JD、笔记等内容拆分为语义片段；
2. 向量化：使用 embedding 模型生成向量；
3. 存储：写入 FAISS 或 Chroma；
4. 检索：根据用户问题或任务检索相关片段；
5. 重排序：优先保留与当前岗位、当前用户任务最相关的内容；
6. 生成：将检索内容作为上下文输入大模型，生成回答。

### 6.4.4 典型应用场景

1. **岗位问答**  
   用户问：“这个岗位为什么适合我？”  
   系统检索该岗位 JD、用户相关项目经历和技能标签，生成匹配解释。

2. **简历优化**  
   用户选择一个岗位，系统检索相关经历，生成针对该岗位的简历 bullet。

3. **面试准备**  
   系统结合 JD 和用户经历，生成高概率面试问题，并给出回答思路。

4. **求职复盘**  
   用户问：“我最近投递失败可能是什么原因？”  
   系统检索历史投递记录、岗位匹配分数、反馈备注，生成分析建议。

## 6.5 模块五：AI 求职助手增强模块

### 6.5.1 功能定位

将 AI 从“文本生成助手”升级为“求职决策助手”。AI 不仅回答问题，还要能主动分析、解释和推荐。

### 6.5.2 核心功能

1. **岗位解读助手**
   - 用简洁语言解释 JD；
   - 提取核心职责、硬性要求、隐性要求；
   - 判断岗位偏产品、运营、技术、销售还是复合型岗位。

2. **投递建议助手**
   - 给出是否建议投递；
   - 解释推荐或不推荐的原因；
   - 标出用户优势和短板；
   - 给出投递前需要补充的材料。

3. **简历改写助手**
   - 根据 JD 选择最相关经历；
   - 生成对应 bullet points；
   - 对已有简历内容进行针对性改写；
   - 输出不同风格版本，如产品岗版、数据岗版、ToB 销售版。

4. **面试模拟助手**
   - 生成岗位相关面试题；
   - 根据用户回答继续追问；
   - 评价回答结构、亮点和不足；
   - 生成 STAR 法则优化版本。

5. **求职策略助手**
   - 根据历史投递数据分析用户更适合的岗位方向；
   - 推荐下一步重点投递类型；

## 9. AI 与数据挖掘的关系设计

本项目中，AI 不应只等同于大模型生成文本，而应拆分为三类能力：

### 9.1 生成式 AI

用于：

- JD 摘要；
- 推荐理由生成；
- 简历 bullet 生成；
- 面试题生成；
- 求职策略报告生成。

### 9.2 检索增强 AI

用于：

- 基于用户真实经历回答岗位相关问题；
- 基于历史投递记录做复盘；
- 基于岗位库解释某类岗位的共同要求；
- 防止简历生成时编造不存在的经历。

### 9.3 机器学习 / 数据挖掘 AI

用于：

- 岗位类型分类；
- 匹配度预测；
- 岗位聚类；
- 技能需求统计；
- 投递转化率分析；
- 个性化岗位排序。

这三类 AI 能力结合后，项目的智能性会明显强于单纯的“AI 生成简历”工具。

## 10. 项目创新点

1. **从岗位记录工具升级为求职决策系统**  
   系统不仅管理投递进度，还能帮助用户判断岗位价值和投递优先级。

2. **引入主动岗位采集能力**  
   相比只依赖用户粘贴 JD，系统能够主动获取岗位信息，形成个人岗位池。

3. **结合数据挖掘与大模型**  
   使用机器学习模型完成岗位分类、匹配预测和聚类分析，再由大模型解释结果。

4. **基于 RAG 的个性化求职助手**  
   AI 回答不再泛泛而谈，而是基于用户简历、项目经历、岗位 JD 和历史投递记录。

5. **形成求职闭环**  
   覆盖岗位采集、岗位筛选、匹配分析、简历优化、面试准备、投递跟踪和求职复盘。

## 13. 风险与应对方案

| 风险 | 表现 | 应对方案 |
|---|---|---|
| 招聘网站反爬 | 页面无法稳定抓取 | 使用插件剪藏、链接解析、模拟数据作为替代 |
| 标注数据不足 | 分类模型效果不稳定 | 使用弱监督标签、少量人工标注、规则兜底 |
| RAG 效果不稳定 | 检索内容不相关 | 优化切片、增加 metadata、按岗位和文档类型过滤 |
| 大模型生成不准确 | 简历内容可能编造 | 要求引用用户经历来源，生成后人工确认 |
| 项目范围过大 | 难以按时完成 | 优先完成 MVP，爬虫和高级模型作为拓展 |
| 前端改动复杂 | 页面整合成本高 | 先做独立页面，再逐步接入原页面 |

## 15. 项目汇报时的核心表达

本项目不是简单地调用大模型生成简历，而是在现有求职管理系统基础上，构建一个完整的智能求职分析系统。系统首先通过采集和导入模块积累岗位数据，再通过结构化解析和技能标签抽取将非结构化 JD 转化为可计算数据；随后利用数据挖掘方法完成岗位分类、匹配度预测和岗位聚类分析；最后结合 RAG 与大模型，为用户提供基于个人经历和真实岗位信息的求职建议、简历优化和面试准备服务。

因此，本项目同时体现了：

- 数据采集；
- 数据清洗；
- 文本结构化；
- 机器学习建模；
- 数据挖掘分析；
- RAG 检索增强生成；
- AI 应用系统开发。

这使项目相比普通求职助手更具课程项目价值和技术完整性。

## 17. 结论

本项目建议以 InternPilot 为基础，不推翻原系统，而是在其“岗位管理 + AI 辅助求职”的框架上进行增强。核心改造方向是：

1. 增加岗位采集和多方式导入，解决数据来源问题；
2. 增加 JD 结构化和技能标签抽取，解决数据可计算问题；
3. 增加岗位分类、匹配预测和聚类分析，体现数据挖掘与模型训练；
4. 增加 RAG 知识库，提升 AI 回答的个性化和真实性；
5. 增强 AI 求职助手，使其真正参与岗位筛选、简历优化、面试准备和求职复盘。

按照该方案开发，项目既能延续原 InternPilot 的实用性，也能明显增强 AI 与数据挖掘成分，适合作为一个完整的课程项目或作品集项目展示。
