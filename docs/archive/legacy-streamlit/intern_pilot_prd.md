# InternPilot 产品需求文档 PRD

## 1. 产品概述

### 1.1 产品名称

**InternPilot：个人化实习推荐与简历适配系统**

英文名称：**InternPilot: Personalized Internship Recommendation and Resume Tailoring System**

---

### 1.2 产品定位

InternPilot 是一个面向个人求职场景的实习岗位推荐、简历适配与面试准备系统。系统通过整合公开实习岗位信息、用户个人经历素材库与面试经验知识库，帮助用户完成从“发现岗位”到“判断岗位是否适合自己”，再到“生成针对性简历经历与面试准备内容”的求职流程。

第一版产品主要服务于个人使用，不做多人平台，不做自动投递，不做复杂招聘平台爬虫，重点解决研究生求职过程中的信息筛选、岗位匹配、简历调整和面试准备问题。

---

### 1.3 目标用户

当前版本的核心用户为：

- 研一或研二学生；
- 正在准备寻找实习；
- 希望同时投递多个方向的岗位；
- 拥有多段科研、数据分析、项目或实践经历，但不知道如何针对不同岗位调整简历；
- 希望用 AI 工具辅助岗位筛选、简历表达和面试准备。

当前项目的第一使用者是用户本人，目标实习方向包括：

1. 数据分析实习；
2. AI 产品 / 产品经理实习；
3. 用户研究实习。

---

### 1.4 核心价值

InternPilot 的核心价值不是“收集尽可能多的岗位”，而是帮助用户完成以下判断：

1. 哪些岗位更适合我？
2. 为什么这个岗位适合我？
3. 我的哪些经历可以用于这个岗位？
4. 简历中应该如何针对该岗位表达这些经历？
5. 面试中可能会被问到什么？
6. 我还需要补充哪些知识点？

产品核心逻辑为：

> 岗位信息结构化 + 个人经历结构化 + 语义匹配 + 大模型生成解释与求职材料。

---

## 2. 产品目标

### 2.1 短期目标：MVP 版本

MVP 版本目标是在较短时间内实现一个可用的个人求职助手，支持：

- 导入或抓取部分实习岗位信息；
- 维护个人经历素材库；
- 设置求职偏好；
- 对岗位进行匹配度评分；
- 推荐高适配岗位；
- 根据岗位 JD 和个人经历生成简历 bullet points；
- 生成面试问题和准备清单；
- 支持围绕岗位、简历和面试进行 AI 问答；
- 可通过 Streamlit / Hugging Face Spaces 在线访问；
- 可作为 GitHub 项目展示。

---

### 2.2 中期目标：展示版

展示版在 MVP 基础上增强项目完整度，支持：

- 页面视觉优化；
- 示例数据；
- 项目 README；
- 系统架构图；
- 数据库结构说明；
- Prompt 设计说明；
- RAG 流程说明；
- 项目截图；
- 可复现部署说明。

---

### 2.3 长期目标：增强版

增强版可进一步支持：

- 多个岗位来源的自动更新；
- 面经向量知识库；
- 投递进度分析；
- 多版本简历管理；
- 根据投递结果优化推荐策略；
- 导出 Word / PDF 简历；
- 模拟面试功能；
- 更完整的岗位推荐算法。

---

## 3. 使用场景

### 3.1 场景一：用户想快速找到适合自己的实习岗位

用户进入系统后，设置目标方向为“数据分析、AI 产品、用户研究”，设置目标城市和公司偏好。系统从已有岗位库中筛选并排序，展示推荐岗位列表。

系统输出：

- 岗位名称；
- 公司；
- 地点；
- 岗位方向；
- 匹配度评分；
- 推荐原因；
- 投递链接；
- 当前状态。

---

### 3.2 场景二：用户看到一个岗位后，想判断是否值得投递

用户点击某个岗位，进入岗位详情页。系统自动分析 JD，并结合用户经历生成岗位适配分析。

系统输出：

- JD 摘要；
- 岗位核心要求；
- 关键词提取；
- 与用户匹配的经历；
- 用户缺少的能力；
- 是否建议投递；
- 推荐理由。

---

### 3.3 场景三：用户想为某个岗位调整简历

用户点击“生成简历经历”按钮。系统根据岗位 JD 和用户经历素材库，推荐适合放入简历的经历，并生成针对该岗位的 bullet points。

系统输出：

- 建议优先展示的 2-3 段经历；
- 每段经历的岗位相关性解释；
- 针对该岗位优化后的简历 bullet points；
- 可以补充强化的技能关键词。

---

### 3.4 场景四：用户想准备面试

用户点击“生成面试准备”按钮。系统结合岗位 JD、岗位类型、公司信息和面经知识库，生成面试准备内容。

系统输出：

- 可能被问到的问题；
- 项目经历追问；
- 技术 / 方法知识点；
- 产品 case 或业务分析题；
- 用户研究方法题；
- 建议补充准备的知识点。

---

### 3.5 场景五：用户想和 AI 聊天，询问岗位相关问题

用户在 AI 求职助手页面选择某个岗位作为上下文，然后向系统提问。

示例问题：

- 这个岗位适合我吗？
- 如果面试官问我为什么想做 AI 产品，我怎么回答？
- 这个数据分析岗位会考 SQL 吗？
- 我的 EEG 项目怎么讲成用户研究经历？
- 这个岗位和我的小红书数据项目有什么关系？
- 请帮我模拟一轮产品经理面试。

---

## 4. 功能需求

## 4.1 用户画像与求职偏好模块

### 4.1.1 功能说明

用户可以设置个人求职偏好，系统根据这些偏好筛选岗位并计算匹配度。

### 4.1.2 字段设计

| 字段 | 类型 | 说明 |
|---|---|---|
| target_roles | 多选 | 目标岗位方向，例如数据分析、AI 产品、用户研究 |
| target_locations | 多选 | 目标城市，例如成都、上海、北京、深圳、远程 |
| preferred_industries | 多选 | 偏好行业，例如互联网、AI、社交媒体、咨询、研究机构 |
| excluded_roles | 多选 | 不考虑岗位，例如销售、客服、强开发岗 |
| internship_duration | 文本 | 可接受实习周期，例如 3 个月以上 |
| available_start_date | 日期 | 可开始实习时间 |
| notes | 文本 | 其他补充偏好 |

### 4.1.3 MVP 优先级

高。

该模块是推荐算法和岗位筛选的基础，MVP 必须实现。

---

## 4.2 个人经历素材库模块

### 4.2.1 功能说明

用户可以将自己的项目、科研、实践和技能经历拆分为结构化素材块。系统后续根据岗位 JD 自动匹配相关经历。

### 4.2.2 经历类型

可包括：

- 科研项目；
- 数据分析项目；
- 产品 / AI 应用项目；
- 用户研究经历；
- 编程项目；
- 论文写作经历；
- 竞赛经历；
- 实习经历；
- 技能模块。

### 4.2.3 用户已有潜在素材

第一版可优先录入以下经历：

1. 小红书人脸 P 图数据采集与互动分析；
2. EEG / LPP / MVPA 审美研究；
3. 眼动 AOI 与 FFL 分析；
4. Buddhist Statues 3D 面部特征分析；
5. 生成式 AI 与审美计算项目；
6. Python / pandas / OpenCV / dlib / 统计建模技能；
7. 英文论文写作与投稿准备经历。

### 4.2.4 字段设计

| 字段 | 类型 | 说明 |
|---|---|---|
| id | 整数 | 经历 ID |
| title | 文本 | 经历名称 |
| type | 文本 | 经历类型 |
| background | 长文本 | 项目背景 |
| methods | 长文本 | 方法与过程 |
| tools | 文本 | 使用工具，如 Python、pandas、dlib、SPSS、MATLAB |
| results | 长文本 | 结果与产出 |
| keywords | 文本 | 关键词 |
| target_roles | 文本 | 适配岗位方向 |
| raw_bullet | 长文本 | 原始简历表述 |
| optimized_bullet | 长文本 | 优化后的通用表述 |
| created_at | 时间 | 创建时间 |
| updated_at | 时间 | 更新时间 |

### 4.2.5 MVP 优先级

高。

这是系统区别于普通岗位搜索工具的核心模块。

## 4.3 岗位采集与岗位库模块

### 4.3.1 功能说明

系统从多个来源导入岗位信息，并转化为统一格式。

### 4.3.2 数据来源优先级

MVP 推荐优先级如下：

1. GitHub 公开实习信息仓库；
2. 目标公司官网招聘页；
3. 手动导入 JD 作为备用入口；
4. 招聘平台链接解析作为后续功能；
5. 大规模招聘平台自动爬取暂不纳入 MVP。

### 4.3.3 岗位字段设计

| 字段 | 类型 | 说明 |
|---|---|---|
| id | 整数 | 岗位 ID |
| company | 文本 | 公司名称 |
| title | 文本 | 岗位名称 |
| location | 文本 | 地点 |
| role_type | 文本 | 岗位类型，例如数据分析、AI 产品、用户研究 |
| source | 文本 | 来源，例如 GitHub、公司官网、手动导入 |
| jd_text | 长文本 | 完整 JD |
| apply_url | 文本 | 投递链接 |
| publish_date | 日期 | 发布时间 |
| deadline | 日期 | 截止时间 |
| skills | 文本 | 技能关键词 |
| status | 文本 | 未查看、感兴趣、已投递、面试中、拒绝、offer |
| match_score | 数值 | 匹配度评分 |
| recommendation_reason | 长文本 | 推荐理由 |
| created_at | 时间 | 创建时间 |
| updated_at | 时间 | 更新时间 |

### 4.3.4 岗位状态

岗位状态包括：

- 未查看；
- 感兴趣；
- 已投递；
- 笔试；
- 面试中；
- 拒绝；
- offer；
- 不合适。

### 4.3.5 MVP 优先级

高。

岗位库是整个系统的入口。

- 地点。

|---|---|

### 4.4.5 MVP 优先级

高。

## 4.5 岗位匹配与推荐模块

### 4.5.1 功能说明

系统根据求职偏好、个人经历素材库和岗位 JD，为岗位计算匹配度，并输出推荐理由。

### 4.5.2 匹配评分维度

| 维度 | 建议权重 | 说明 |
|---|---:|---|
| 岗位方向匹配 | 30% | 是否属于数据分析、AI 产品、用户研究 |
| 技能匹配 | 25% | JD 技能要求与用户技能是否匹配 |
| 经历匹配 | 25% | 用户项目经历是否能支撑岗位要求 |
| 地点匹配 | 10% | 是否符合目标城市 |
| 投递价值 | 10% | 公司质量、岗位成长性、是否适合研一 |

### 4.5.3 推荐等级

| 分数 | 推荐等级 | 含义 |
|---:|---|---|
| 85-100 | 强烈推荐 | 高度适配，应优先投递 |
| 70-84 | 推荐 | 适合投递 |
| 55-69 | 谨慎推荐 | 可投递，但需要补充准备 |
| 0-54 | 暂不优先 | 当前不建议优先投递 |

### 4.5.4 推荐理由生成

系统应输出简洁、可解释的推荐理由，例如：

> 该岗位要求 Python、数据分析和用户行为分析，与用户的小红书数据采集与互动建模项目高度匹配。同时，岗位强调内容平台理解，用户已有社交媒体数据研究经验，因此推荐优先投递。

### 4.5.5 MVP 实现方式

MVP 可采用：

1. 规则打分；
2. 关键词匹配；
3. Embedding 相似度；
4. LLM 生成推荐解释。

### 4.5.6 MVP 优先级

高。

这是产品核心功能。

## 4.6 简历经历生成模块

### 4.6.1 功能说明

用户选择某个岗位后，系统根据该岗位 JD 和用户经历素材库，推荐最适合放入简历的经历，并生成针对该岗位的简历 bullet points。

### 4.6.2 输入

- 岗位 JD；
- 岗位类型；
- 岗位关键词；
- 用户经历素材库；
- 用户求职方向。

### 4.6.3 输出

- 推荐展示的经历；
- 每段经历与岗位的匹配理由；
- 针对岗位优化后的简历 bullet points；
- 建议补充的技能关键词；
- 不建议强调的内容。

### 4.6.4 输出示例：数据分析岗位

- 使用 Python、pandas 对社交媒体图文数据进行清洗、特征提取与统计建模，围绕点赞、收藏、评论等互动指标构建分析框架。
- 基于负二项回归模型分析面部编辑程度与用户互动之间的非线性关系，识别影响内容传播效果的关键视觉特征。
- 结合可视化方法输出数据分析报告，为平台内容策略、视觉偏好分析和用户行为理解提供依据。

### 4.6.5 输出示例：AI 产品岗位

- 围绕 AIGC 审美计算场景，梳理用户视觉偏好、审美判断与产品化应用路径，探索认知神经数据在 AI 产品评估中的转化价值。
- 基于 EEG 与行为评分数据，分析用户对不同类型面部编辑内容的审美加工机制，为 AI 审美评估与用户感知建模提供数据依据。
- 结合社交媒体图像编辑数据与用户互动指标，分析用户对视觉内容生成与编辑功能的偏好，为 AI 图像产品功能优化提供参考。

### 4.6.6 输出示例：用户研究岗位

- 设计并分析面部审美相关实验，结合 EEG、眼动和主观评分数据，研究用户对不同视觉特征的注意分配与审美判断机制。
- 基于 AOI 区域划分和眼动指标，分析用户在面部不同区域的视觉关注模式，为理解用户感知和产品视觉体验提供依据。
- 结合行为评分与多模态数据，提炼用户对面部编辑内容的感知差异与潜在审美偏好。

### 4.6.7 MVP 优先级

高。

这是用户最直接需要的功能之一。

## 4.7 面试准备模块

### 4.7.1 功能说明

用户选择岗位后，系统根据 JD、岗位类型、用户经历和面经知识库，生成面试准备内容。

### 4.7.2 输出内容

- 可能面试问题；
- 简历项目追问；
- 技术知识点；
- 业务分析题；
- 产品 case；
- 用户研究方法题；
- 需要补充准备的内容；
- 建议回答思路。

### 4.7.3 不同岗位方向的面试准备重点

#### 数据分析岗位

重点包括：

- Python / SQL；
- 数据清洗；
- 指标体系；
- 统计建模；
- A/B 测试；
- 业务分析；
- 可视化表达；
- 项目中的数据来源与建模解释。

#### AI 产品岗位

重点包括：

- AI 产品理解；
- 用户需求分析；
- 功能设计；
- Prompt / 大模型应用；
- 产品指标；
- 竞品分析；
- 技术边界理解；
- 如何将研究经历转化为产品思维。

#### 用户研究岗位

重点包括：

- 用户访谈；
- 问卷设计；
- 实验设计；
- 定性与定量研究方法；
- 用户体验评估；
- 行为数据分析；
- 眼动 / EEG 项目的用户研究表达方式。

### 4.7.4 MVP 优先级

中高。

MVP 中应实现基础面试问题生成；面经 RAG 可作为后续增强。

## 4.9 AI 求职助手模块

### 4.9.1 功能说明

用户可以与 AI 进行岗位、简历和面试相关对话。AI 回答时应优先使用当前岗位 JD、用户经历素材库和面经知识库作为上下文。

### 4.9.2 支持问题类型

- 岗位适配判断；

## 7. 推荐算法设计

### 7.1 总体思路

MVP 阶段采用轻量级推荐算法，不训练复杂模型。推荐逻辑由三部分组成：

1. 规则匹配；
2. 关键词匹配；
3. 语义相似度匹配；
4. 大模型生成解释。

---

### 7.2 匹配评分公式

```text
match_score = role_score * 0.30
            + skill_score * 0.25
            + experience_score * 0.25
            + location_score * 0.10
            + value_score * 0.10
```

其中：

- role_score：岗位方向匹配分；
- skill_score：技能关键词匹配分；
- experience_score：个人经历与 JD 的相似度；
- location_score：城市匹配分；
- value_score：投递价值分。

---

### 7.3 规则示例

#### 岗位方向匹配

- JD 或岗位名称包含“数据分析、商业分析、数据运营、用户分析”：归为数据分析；
- 包含“产品经理、AI 产品、AIGC、策略产品”：归为 AI 产品；
- 包含“用户研究、UX Research、用研、用户体验研究”：归为用户研究。

#### 技能关键词匹配

数据分析关键词：

- Python；
- SQL；
- pandas；
- Excel；
- 统计分析；
- 数据可视化；
- A/B test；
- 指标体系；
- 用户行为分析。

AI 产品关键词：

- AIGC；
- 大模型；
- Prompt；
- 产品设计；
- 需求分析；
- 竞品分析；
- 用户反馈；
- AI 应用。

用户研究关键词：

- 用户访谈；
- 问卷；
- 实验设计；
- 眼动；
- EEG；
- 行为数据；
- 用户体验；
- 定性研究；
- 定量研究。

## 11. 非功能需求

### 11.1 可用性

- 页面应简洁，适合个人快速使用；
- 重要按钮应清晰，例如“生成简历经历”“生成面试准备”；
- 推荐结果应可解释，不能只给分数。

### 11.2 数据安全

- 用户简历和个人经历数据应本地保存或私有部署；
- API key 不应上传到 GitHub；
- 不采集他人隐私信息；
- 不自动抓取需要登录或权限受限的内容。

### 11.3 合规性

- MVP 阶段不进行大规模招聘平台爬虫；
- 优先使用公开 GitHub 仓库、公司官网和用户手动导入内容；
- 对招聘平台内容仅作为用户主动输入后的单条解析，不做批量采集。

### 11.4 可扩展性

- 数据库表结构应方便后续增加字段；
- LLM prompt 应独立存放，便于调整；
- 岗位采集模块应与匹配模块解耦；
- 后续可将 SQLite 替换为 PostgreSQL。

## 13. 风险与应对

### 13.1 岗位来源不稳定

风险：公司官网页面结构变化，GitHub 仓库更新不稳定。

应对：

- 保留手动导入入口；
- 优先支持结构简单的数据源；
- 将采集模块与主系统解耦。

---

### 13.2 大模型生成内容不准确

风险：生成的简历内容可能夸大或编造经历。

应对：

- Prompt 中明确要求“不编造”；
- 生成内容后用户确认再使用；
- 输出中保留对应经历来源。

---

### 13.3 项目开发范围过大

风险：功能过多，导致下个月前无法完成。

应对：

- 第一版只做个人使用；
- 不做用户系统；
- 不做自动投递；
- 不做 PDF 简历导出；
- 不做大规模招聘平台爬虫；
- 优先保证岗位推荐、简历生成和面试准备三个核心功能。

---

### 13.4 合规风险

风险：大规模抓取招聘平台信息可能违反平台规则。

应对：

- 第一版不做招聘平台批量爬虫；
- 优先使用公开信息源；
- 用户主动输入的单条 JD 可以解析；
- 不采集个人隐私数据。
